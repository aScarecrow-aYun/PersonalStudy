# 生产者-消费者队列

- 为什么需要它？假设我们在写一个日志系统：
  
  - **业务线程（生产者）**：处理用户请求，顺手记个日志 `logger.log("处理成功")`。处理请求极快，写硬盘极慢。
  
  - **写盘线程（消费者）**：专门负责把日志写到硬盘上。

- **反面教材**（不使用队列）：业务线程直接调用`write(disk)`，由于写硬盘慢，业务线程被卡住（阻塞），导致用户请求超时。如果开100个业务线程同时写日志，硬盘可能被并发读写搞崩溃。

- **痛点**：
  
  1. **速度不对等**：快生产者 vs 慢消费者（或者快消费者 vs 慢生产者）。
  
  2. **强耦合**：生产者必须等待消费者干完活才能继续。

- **核心思想**：以队列作为“缓冲池”
  
  > **生产者-消费者模式**：在生产和消费之间加上一个**线程安全的队列（缓冲区）**。
  > 
  > - **生产者**：只管往队列里扔数据，扔完立刻走人，绝不阻塞业务。
  > 
  > - **消费者**：专门有个线程死循环盯着队列，有数据就拿出来处理，没数据就睡觉。
  > 
  > - **解耦神效**：生产者根本不知道消费者是谁、有几个、在不在运行。

```cpp
#pragma once
#include <iostream>
#include <sstream>
#include <fstream>
#include <filesystem>
#include <queue>
#include <vector>
#include <mutex>
#include <atomic>
#include <thread>
#include <condition_variable>
#include <format>
#include <functional>
#include <tuple>

#ifdef _WIN32
#include <windows.h>
#else
#include <unistd.h>
#endif

namespace fs = std::filesystem;

inline std::string get_timestamp() {
    using namespace std::chrono;
    const auto now = system_clock::now();
    const auto t = system_clock::to_time_t(now);

    std::tm tm_info;
#ifdef _WIN32
    localtime_s(&tm_info, &t);
#else
    localtime_r(&t, &tm_info);
#endif

    const auto ms = duration_cast<milliseconds>(now.time_since_epoch()) % 1000;
    char buffer[30];
    std::strftime(buffer, sizeof(buffer), "%Y-%m-%d %H:%M:%S", &tm_info);
    return std::string(buffer) + "." + std::to_string(ms.count());
}

class Logger {
public:
    static Logger& instance() {
        static Logger logger;
        return logger;
    }

    Logger(const Logger&) = delete;
    Logger& operator=(const Logger&) = delete;

    enum class LogLevel { Info, Warning, Error, Debug };

    template<typename... Args>
    void Log(LogLevel level, const std::string& format, Args&&... args) {
        // 将格式化字符串和所有参数打包成一个 Lambda 表达式（闭包）
        // 注意：必须使用std::decay_t按值拷贝参数！如果传引用，后台线程执行时引用可能已经失效（悬垂引用）
        auto formatter = [f = format, t = std::make_tuple(std::decay_t<Args>(args)...)]() mutable {
            // 在后台线程中被调用时，才真正执行 vformat
            return std::apply([&f](auto&&... a) {
                return std::vformat(f, std::make_format_args(a...));
                }, t);
            };

        // 业务线程只需要塞入这个 Lambda，极其迅速，不消耗CPU做字符串拼装
        {
            std::lock_guard<std::mutex> lock(m_mtxQ);
            m_logQueue.push({ level, std::this_thread::get_id(), std::move(formatter) });
        }

        m_condQ.notify_one();
    }

    void Stop() {
        if (m_running) {
            m_running = false;
            m_condQ.notify_all();
            if (m_workerThread.joinable()) {
                m_workerThread.join();
            }
        }
    }

private:
    // 智能刷新的策略配置（可根据业务需求调整）
    static constexpr int kMaxUnflushedLogs = 50;         // 攒够50条刷一次
    static constexpr int kMaxFlushIntervalSec = 3;       // 或者最多3秒刷一次

    Logger() { Init(fs::current_path() / "Log"); }
    ~Logger() {
        Stop();
        if (m_logFile && m_logFile.is_open()) {
            m_logFile.close();
        }
    }

    void Init(const fs::path& basePath) {
        m_basePath = basePath;
        m_running = true;
        m_workerThread = std::thread(&Logger::WorkerThread, this);
    }

    void WorkerThread() {
        using namespace std::chrono_literals;
        std::vector<LogEntry> local_entries;

        int unflushed_count = 0;
        auto last_flush_time = std::chrono::steady_clock::now();

        while (m_running) {
            {
                std::unique_lock<std::mutex> lock(m_mtxQ);

                // 如果队列为空，wait_for会执行以下动作：
                // 1. 检查Lambda表达式的返回值状态：发现条件不满足（Lambda返回false），决定去“睡觉”
                // 2. 自动解锁：wait_for会自动调用lock.unlock()去解锁
                // 3. 去“睡觉”：线程挂起，准备睡100ms
                // 4. 在这100ms内，互斥锁是没有被当前线程占用的，也就是说此时可以不断地抢锁、入队
                // 5. 接着线程被唤醒（要么是100ms超时时间到了，要么是业务线程提前叫醒）
                // 6. 苏醒，在wait_for函数真正return返回之前，它会自动调用lock.lock()
                // 7. 抢到锁后，它会再次检查Lambda表达式
                m_condQ.wait_for(lock, 100ms, [this] {
                    return !m_logQueue.empty() || !m_running;
                    });

                while (!m_logQueue.empty()) {
                    local_entries.push_back(std::move(m_logQueue.front()));
                    m_logQueue.pop();
                }
            }

            for (const auto& entry : local_entries) {
                try {
                    // 在这里才真正调用Lambda进行格式化
                    std::string message = entry.formatter();

                    const auto now = std::chrono::system_clock::now();
                    const auto currentHour = std::chrono::duration_cast<std::chrono::hours>(now.time_since_epoch());

                    if (!m_logFile.is_open() || currentHour != m_currentHour) {
                        // 在切换日志文件前，必须把旧文件里的缓存强制刷入磁盘
                        FlushLogFile();
                        RotateLogFile();
                        m_currentHour = currentHour;
                    }

                    std::ostringstream oss;
                    oss << "[" << get_timestamp() << "]["
                        << LogLevelToString(entry.level) << "]["
                        << entry.thread_id << "]" << message;

                    if (m_logFile.is_open()) {
                        m_logFile << oss.str() << "\n";
                        unflushed_count++;
                    }
                }
                catch (...) {
                    // 捕获格式化异常，防止后台线程崩溃
                    std::cerr << "Logger format/write exception occurred!" << std::endl;
                }
            }

            // 智能判断是否需要刷新到硬盘
            auto current_time = std::chrono::steady_clock::now();
            auto time_since_last_flush = std::chrono::duration_cast<std::chrono::seconds>(current_time - last_flush_time);

            if (unflushed_count >= kMaxUnflushedLogs || time_since_last_flush.count() >= kMaxFlushIntervalSec) {
                FlushLogFile();
                unflushed_count = 0;
                last_flush_time = current_time;
            }

            local_entries.clear();
        }

        // 程序结束时，如果缓冲区里还有没刷盘的日志，强制全部刷入硬盘，防止丢失
        if (unflushed_count > 0) {
            FlushLogFile();
        }
    }
=
    void FlushLogFile() {
        if (m_logFile.is_open()) {
            m_logFile.flush();
        }
    }

    void RotateLogFile() {
        if (m_logFile.is_open()) {
            m_logFile.close();
        }
        const fs::path logPath = GetCurrentLogPath();
        fs::create_directories(logPath.parent_path());

        m_logFile.open(logPath, std::ios::app);
        if (!m_logFile.is_open()) {
            std::cerr << "Unable to open the log file: " << logPath.string() << std::endl;
        }
    }

    fs::path GetCurrentLogPath() const {
        const auto now = std::chrono::system_clock::now();
        const std::time_t t = std::chrono::system_clock::to_time_t(now);
        std::tm tm;
#ifdef _WIN32
        localtime_s(&tm, &t);
#else
        localtime_r(&t, &tm);
#endif

        std::ostringstream dateDirOSS;
        dateDirOSS << std::put_time(&tm, "%Y-%m-%d");
        fs::path dateDir = m_basePath / dateDirOSS.str();

        std::ostringstream fileNameOSS;
        fileNameOSS << std::put_time(&tm, "%Y-%m-%d_%H.log");
        return dateDir / fileNameOSS.str();
    }

    std::string LogLevelToString(const LogLevel& level) const {
        switch (level) {
        case LogLevel::Info:    return "Info";
        case LogLevel::Warning: return "Warning";
        case LogLevel::Error:   return "Error";
        case LogLevel::Debug:   return "Debug";
        default:                return "Info";
        }
    }

private:
    std::atomic<bool> m_running{ false };
    std::thread m_workerThread;

    struct LogEntry {
        LogLevel level;
        std::thread::id thread_id;
        // 不再存string，而是存一个“可调用对象”（延迟执行的函数）
        std::function<std::string()> formatter;
    };

    std::queue<LogEntry> m_logQueue;
    mutable std::mutex m_mtxQ;
    std::condition_variable m_condQ;

    fs::path m_basePath;
    std::ofstream m_logFile;
    std::chrono::hours m_currentHour;
};
```

---

**代码解析**

1. **Q**：你在后台消费线程里用了`unique_lock`，而在业务线程 `push` 时用了`lock_guard`，为什么要有这种区分？能都用`lock_guard`吗？
   
   **A** 绝对不能都用（原因在代码注释里，这里略）

2. **Q**：为什么用`wait_for`而不是用`wait`？
   
   **A**：如果用`wait`，线程会“死等”，直到有人叫它才醒
   
   - 原因一：死等的话就没机会去执行刷盘策略
   
   - 原因二：哪怕因为操作系统调度的原因导致`notify()`没被调用或失效，这里100ms后线程自动醒来，看到`m_running == false`，就能够自动安全跳出循环结束线程

3. **Q**： `cv.wait_for()` 里面加了一个Lambda表达式作为断言，这是为了防什么？如果不加会怎样？
   
   **A**：这是为了防止虚假唤醒。虚假唤醒指的是线程莫名其妙地醒了，但实际上并没有人叫醒它，且条件也没有满足。在Linux等操作系统的底层实现中，线程偶尔会没有任何原因就莫名其妙地从`wait`中醒来。如果不加Lambda断言：`m_condQ.wait_for(lock, 100ms);`，醒来后不管队列是不是空的，直接执行`queue.front()`就会产生未定义行为甚至段错误崩溃。加上Lambda表达式后，即使虚假醒来了，它也会重新检查条件，发现队列为空就继续滚回去睡觉，保证了绝对的线程安全。

---

**亮点**：上述日志类代码有如下亮点：

1. 延迟格式化
   
   - 业务线程调用 `Log()` 时，绝不拼接字符串。它利用 C++20 的Lambda表达式和`std::tuple`将“格式化规则”和“参数副本”打包成一个闭包（`std::function`）塞进队列
   - 真正的字符串拼装（`std::vformat`）被推迟到后台日志线程中进行。这实现了业务线程的零内存分配、极致低延迟
   - 巧妙使用`std::decay_t`按值捕获参数，完美解决了跨线程传递参数可能引发的“悬垂引用”崩溃问题

2. 智能异步刷盘
   
   - 摒弃了低效的“每写一条就 `flush()`”的策略
   
   - 引入了双阈值触发”机制：攒够 50 条（`kMaxUnflushedLogs`）或者距离上次刷盘超过 3 秒（`kMaxFlushIntervalSec`）才真正执行一次 I/O 刷新。这极大地提升了磁盘 I/O 的吞吐量

3. 并发安全
   
   - RAII 批量无锁处理架构：后台线程使用`unique_lock`配合作用域`{}`，一旦批量偷取完队列数据，立刻自动解锁。后续的写文件、格式化过程完全在无锁状态下进行，既不会死锁，也不会阻塞生产者
   
   - 异常隔离（防宕机）：格式化和写入过程被`try-catch`包裹。即使业务代码传入了无法格式化的奇葩参数，也只会吞掉异常并打印到`stderr`，绝不会导致后台日志线程崩溃，从而保护主进程不宕机
   
   - 无数据丢失保障：
     
     - 在跨越小时切换日志文件（`RotateLogFile`）前，强制调用 `FlushLogFile()`
     
     - 在程序退出（`Stop()` 结束循环）时，检查缓冲区如果还有残余日志，强制执行最后一次 `FlushLogFile()`，做到了优雅退出

4. 不依赖`fmt`等第三方库，完全依托现代 C++ 标准库构建
