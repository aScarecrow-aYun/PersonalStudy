# 单例模式

## 定义

单例模式保证一个类仅有一个实例，并提供一个全局访问点来访问该实例。

---

## 使用场景

- 日志记录器

- 数据库连接池、线程池

- 系统配置管理器（用于读取全局配置文件）

---

## 实现关键

1. **私有化构造函数**

2. **删除拷贝构造函数和赋值运算符**

3. **提供静态公有访问方法**

---

## 常见实现方式

### 方式一：懒汉式（线程不安全，但改进后的懒汉式是线程安全且高效的）

- 特点：第一次调用对象时才创建对象

- 缺陷：线程不安全。在多线程环境下，如果多个线程同时执行到`if (instance == nullptr)`，会创建出多个实例
  
  > 为什么懒汉式是线程不安全的？核心在于这句代码：`instance = new Singleton();`
  > 
  > `new`操作不是原子操作，而`new`被放在了`getInstance()`函数里，这意味着对象的构建被延迟到了程序运行期间（也就是`main()`函数执行之后）
  > 
  > 假设A和B两个线程同时调用该函数：
  > 
  > - 时刻1：线程A读取`instance`，发现是`nullptr`，准备进入`if`
  > 
  > - 时刻2：操作系统把CPU切换给线程B，线程B读取`instance`，发现也是`nullptr`，也准备进入`if`
  > 
  > - 时刻3：线程 A 执行 `new`，得到了一个地址，比如 `0x1111`，赋值给 `instance`。
  > 
  > - 时刻4：线程 B 执行 `new`，得到了另一个地址，比如 `0x2222`，也赋值给 `instance`。
  > 
  > - 结果：`instance`最后指向了`0x2222`，但`0x1111`那块内存泄漏了，单例被破坏了。

```cpp
class Singleton {
private:
    Singleton() {}
    ~Singleton() {}
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

    static Singleton* instance;
public:
    static Singleton* getInstance() {
        if (instance == nullptr) {
            instance = new Singleton();
        }
        return instance;
    }
};

// 初始化
Singleton* Singleton::instance = nullptr;
```

我们可以通过加锁的方式来保证懒汉式的线程安全：

```cpp
#include <iostream>
#include <mutex>

class Singleton {
private:
    static Singleton* instance;
    static std::mutex mtx; // 静态互斥锁

    Singleton() { std::cout << "Singleton created.\n"; }
    ~Singleton() {}
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

public:
    static Singleton* getInstance() {
        mtx.lock(); // 1. 进门前加锁

        if (instance == nullptr) {
            instance = new Singleton();
        }

        mtx.unlock(); // 2. 出门前解锁
        return instance;
    }
};

// 类外初始化
Singleton* Singleton::instance = nullptr;
std::mutex Singleton::mtx;
```

但上述代码有个致命的缺点：无论`instance`有没有被初始化，每次调用`getInstance()`都要去抢锁、解锁。让成百上千个线程为了一个普通的读操作去排队，会导致程序并发性能急剧下降。

所以衍生出了下面这种更优的加锁方式，称之为“双重检查锁定模式`DCLP(Double Checked Locking Pattern)`”：

```cpp
static Singleton* getInstance() {
    // 第一次检查：如果已经初始化，直接返回，避免无谓的加锁开销
    if (instance == nullptr) { 

        mtx.lock();

        // 第二次检查：抢到锁后再次确认，确保只有一个线程能 new
        if (instance == nullptr) { 
            instance = new Singleton();
        }

        mtx.unlock();
    }
    return instance;
}
```

上述代码理论上是最完美的，但实际运行中依然有问题，原因在于`instance = new Singleton();`这一句实际上包含了三个步骤：

1. 分配内存空间

2. 调用构造函数，在内存上构造`Singleton`对象

3. 将`instance`指针指向这块内存

编译器为了优化性能，可能会把这三步的顺序打乱，例如变成1->3->2。

假设线程A执行了1和3（此时`instance`已经不是`nullptr`了，但对象还没构造），此时CPU切换到线程B

线程B执行第一次检查，发现`instance != nullptr`，直接返回了这个指针去使用：**访问了未构造的内存，程序崩溃**

C++11引入了内存序和`std::atomic`，允许我们手动解决指令重排的问题。我们需要把指针声明为原子变量：

```cpp
#include <atomic>
#include <mutex>

class Singleton {
private:
    // 使用 atomic 解决指令重排问题
    static std::atomic<Singleton*> instance; 
    static std::mutex mtx;

    Singleton() {}
    ~Singleton() {}
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

public:
    static Singleton* getInstance() {
        // memory_order_acquire：保证在这里之后的代码不会被重排到这之前
        Singleton* tmp = instance.load(std::memory_order_acquire); 
        if (tmp == nullptr) {
            std::lock_guard<std::mutex> lock(mtx); // RAII 方式加锁，更安全
            tmp = instance.load(std::memory_order_relaxed);
            if (tmp == nullptr) {
                tmp = new Singleton();
                // memory_order_release：保证 new 的所有操作都在赋值之前完成
                instance.store(tmp, std::memory_order_release); 
            }
        }
        return tmp;
    }
};
```

这段代码在多线程环境下是绝对安全的，且性能极佳（只有在第一次初始化时加锁），但是这代码太复杂了。

于是C++11干脆提供了一个专门干这个的工具：`std::call_once()`。它保证无论多少个线程同时调用它，传入的函数只会被执行一次：

```cpp
#include <mutex>

class Singleton {
private:
    static Singleton* instance;
    static std::once_flag flag; // 配合 call_once 使用的标志位

    Singleton() { std::cout << "Singleton created.\n"; }
    ~Singleton() {}
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

public:
    static Singleton* getInstance() {
        // 无论多少个线程并发调用这一句，init 函数只会被执行一次
        std::call_once(flag, []{
            instance = new Singleton();
        });
        return instance;
    }
};

// 初始化
Singleton* Singleton::instance = nullptr;
std::once_flag Singleton::flag;
```

### 方式二：饿汉式（线程安全 - 可用但不完美，❌️不推荐）

- 特点：在程序启动时就完成实例化

- 优点：天然线程安全
  
  > 为什么饿汉式是线程安全的？核心在于这一句代码`Singleton Singleton::instance;`
  > 
  > 在C++中，全局变量和静态成员变量是在程序进入`main()`函数之前就被初始化的
  > 
  > 程序的启动过程大致是这样的：
  > 
  > 1. 操作系统加载可执行文件
  > 
  > 2. 分配内存，初始化全局变量和静态变量（此时是单线程的，只有主线程在跑）
  > 
  > 3. 调用`main()`函数，之后在`main()`函数中你可能会创建多个线程
  > 
  > 也就是说，在程序进入多线程环境时，`Singleton::instance`这个对象早就已经构造完成了
  > 
  > 所以当多线程后来调用`getInstance()`时，函数内部只有一句话：`return instance`

- 缺点：如果单例对象构造非常耗时，或者程序运行过程中根本没用到它，会导致启动变慢和资源浪费

```cpp
class Singleton {
private:
    Singleton() {}
    ~Singleton() {}
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

    static Singleton instance; // 直接声明静态对象
public:
    static Singleton& getInstance() {
        return instance;
    }
};

// 在类外进行定义和初始化
Singleton Singleton::instance;

// =================================================================
// 另一种写法
class Singleton {
private:
    Singleton() {}
    ~Singleton() {}
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

    static Singleton* instance;
public:
    static Singleton* getInstance() {
        return instance;
    }
};

Singleton* Singleton::instance = new Singleton();
```

### 方式三：局部静态变量（Meyer's Singleton - ✅️最推荐）

- 特点：在C++11标准中明确规定：如果局部静态变量在第一次执行到声明处时未被初始化，那么即使多个线程并发执行到这一行，编译器也会在底层暗中加锁，保证构造函数只被执行一次。
  
  注意静态局部变量是在“程序的控制流第一次执行到它的声明语句时”被初始化的，而全局变量、全局/类静态变量是在`main()`函数执行前就被初始化了。

- 优点：
  
  1. 绝对线程安全
  
  2. 懒加载
  
  3. 代码简洁
  
  4. 无需加锁，性能高

```cpp
class Singleton {
private:
    Singleton() { std::cout << "Singleton created.\n"; }
    ~Singleton() {}

    // 禁用拷贝构造（用已有的对象来“克隆”出一个新对象）
    // 破坏单例的写法：
    // Singleton& s1 = Singleton::getInstance();
    // Singleton s2(s1);
    Singleton(const Singleton&) = delete;

    // 禁用拷贝赋值
    // 破坏单例的写法：
    // Singleton& s1 = Singleton::getInstance();
    // Singleton s2;//实际上由于构造函数设为了private，这样写是会编译报错的，但显示delete它是为了保持语法严谨（防御性编程）
    // s2 = s1;
    Singleton& operator=(const Singleton&) = delete;

    // 禁止移动构造
    // 破坏单例的写法：
    // Singleton s2(std::move(Singleton::getInstance()));
    // 破坏的原因在于调用了移动构造，把单例内部的资源移动给了s2，单例变成了空壳，s2成了新对象
    Singleton(Singleton&&) = delete;

    // 禁止移动赋值
    // 破坏单例的写法：
    // Singleton s2;
    // s2 = std::move(Singleton::getInstance());
    Singleton& operator=(Singleton&&) = delete;

    // 为什么饿汉和懒汉的常见代码里只有前两个？
    // 这是因为在C++11之前（C++98/03）根本没有移动语义，也就没有后两个函数（移动构造和移动赋值）
    // 那时候防止对象被复制，只需要防止拷贝构造和拷贝赋值这两个即可
    // 在C++11及之后，引入了右值引用和移动语义。按理说，既然有了新途径能创建对象，单例就必须把新途径也delete掉
    // 那为什么很多人不写后面两个，代码依旧能编译通过且安全呢？
    // 这是因为C++11的一个规则：如果你显式声明了拷贝构造函数或拷贝赋值运算符（无论是自己写、设为 = default 还是 = delete），编译器就不再自动为你生成移动构造和移动赋值函数了
    // 那既然只写前面两个就安全，为什么Modern C++推荐写全四个？
    // 这是因为显示地delete掉移动构造和移动赋值函数是更好的编程规范，任何读代码的人（比如说新手，或者是不清楚这个规则的人）看到这四个 = delete，都能够瞬间明白

public:
    static Singleton& getInstance() {
        // 局部静态变量，C++11 保证线程安全
        static Singleton instance; 
        return instance;
    }
};
```

如果有多个类都需要实现单例，为了避免重复写上述的代码，我们可以封装一个单例模板基类：

```cpp
template <typename T>
class Singleton {
protected:
    Singleton() = default;
    virtual ~Singleton() = default;

    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;
    Singleton(Singleton&&) = delete;
    Singleton& operator=(Singleton&&) = delete;

public:
    static T& getInstance() {
        static T instance;
        return instance;
    }
};

// 使用示例：
class Logger : public Singleton<Logger> {
    // 必须声明为友元，否则基类无法调用私有构造
    // 为什么？当我们调用Logger::getInstance()时，由于Logger没有写这个方法，它会去父类Singleton<Logger>找
    // 当执行到static T instance这一句时，因为T就是Logger，编译器需要在这里构造一个Logger对象，也就是需要调用Logger()这个构造函数
    // 此时矛盾爆发了：
    // 我们为了让外面不能直接new Logger()，把Logger()设为了private
    // 现在去调用它的，是父类Singleton<Logger>
    // C++编译器一看，调用者是Singleton<Logger>，被调用者是Logger的private函数，权限不够，然后就报错了
    // 在编译器眼里，不管你是不是我的父类，我（子类）的private你不能访问
    // 所以这里用friend来打破权限限制，让父类能够调用子类的私有属性和方法
    friend class Singleton<Logger>;
private:
    Logger() { std::cout << "Logger initialized.\n"; }
public:
    void log(const std::string& msg) {
        std::cout << "[LOG] " << msg << std::endl;
    }
};

// 调用：
// Logger::getInstance().log("System started");
```

## 注意事项

单例模式只保证了一件事：全局只有这一个实例，并且这个实例的创建过程是线程安全的，它完全不保证这个实例被创建出来之后，上百个线程同时使用它时的安全性。

如果单例有写操作（如日志类、连接池类），必须在单例类内部的业务方法中进行加锁保护，例如下面的代码：

```cpp
class Logger {
private:
    std::string buffer;
    std::mutex log_mtx; // 单例内部拥有一把锁
    Logger() {}
public:
    static Logger& getInstance() {
        static Logger instance;
        return instance;
    }

    // 使用时加锁
    void log(const std::string& msg) {
        std::lock_guard<std::mutex> lock(log_mtx); // 保护具体的业务逻辑
        buffer += msg;
        // 写入文件...
    }
};
```
