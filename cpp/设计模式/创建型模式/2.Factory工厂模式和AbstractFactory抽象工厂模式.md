# 工厂模式

## 为什么需要工厂模式

在面向对象编程中，我们最常做的事就是 `new` 一个对象。但直接 `new` 有一个致命缺点：**客户端代码与具体类死死绑定在了一起（耦合度高）**。

- **反面教材**（不用工厂）：

```cpp
// 假设你在做一个跨平台 UI 框架，需要创建按钮，业务代码中到处都是这样的硬编码
void drawUI() {
    if (platform == "Windows") {
        Button* btn = new WinButton(); // 依赖了具体的 WinButton 类
    } else if (platform == "Mac") {
        Button* btn = new MacButton(); // 依赖了具体的 MacButton 类
    }
    btn->draw();
}
```

- **痛点**：如果以后要增加`LinuxButton`，或者把`WinButton`换成`WinButtonPro`，那么必须修改`drawUI`的`if-else`。假设项目里有 100 个地方`new`了按钮，你就得改100处。这严重违反了**开闭原则（OCP：对扩展开放，对修改关闭）**。

- **核心思想**：不要让业务代码去`new`对象，把`new`的动作剥离出来，交给专门的“工厂”去干。

## 一、简单工厂模式

- **注意**：简单工厂不属于23种设计模式之一，但它是理解工厂模式的必经之路。

- **核心思想**：建一个单一的工厂类，里面写一个静态方法，根据传入的参数（字符串、枚举等），决定`new`哪个具体的对象。

- **优点**：客户端脱离了具体产品的创建细节，只需要传个参数就行。

- **缺点**：**违反了开闭原则**。如果我要加一个`LinuxButton`，我必须打开`ButtonFactory`的源码，去修改`createButton`里的`if-else`分支。

```cpp
#include <iostream>
#include <memory>
#include <string>

// 1. 抽象产品接口
class Button {
public:
    virtual ~Button() = default;
    virtual void draw() const = 0;
};

// 2. 具体产品
class WinButton : public Button {
public:
    void draw() const override { std::cout << "绘制 Windows 风格按钮\n"; }
};

class MacButton : public Button {
public:
    void draw() const override { std::cout << "绘制 Mac 风格按钮\n"; }
};

// 3. 简单工厂类
class ButtonFactory {
public:
    // 静态方法，返回智能指针
    static std::unique_ptr<Button> createButton(const std::string& type) {
        if (type == "Windows") {
            return std::make_unique<WinButton>();
        } else if (type == "Mac") {
            return std::make_unique<MacButton>();
        }
        return nullptr; // 未知类型
    }
};

// 4. 客户端使用
int main() {
    // 客户端不再依赖具体的 WinButton/MacButton，只依赖 Button 接口和 Factory
    auto btn1 = ButtonFactory::createButton("Windows");
    auto btn2 = ButtonFactory::createButton("Mac");

    if (btn1) btn1->draw();
    if (btn2) btn2->draw();
    // 不需要手动 delete，unique_ptr 出作用域自动释放
    return 0;
}
```

## 二、Factory工厂方法模式（23种设计模式之一）

- 为了解决简单工厂“修改源码”的问题，工厂方法来了：**不要只搞一个全能工厂，把“工厂”也抽象化！**

- **核心思想**：定义一个用于创建对象的**接口**（抽象工厂），但让子类决定实例化哪个类。工厂方法让类把实例化推迟到子类。

- **优点**：完美符合开闭原则
  
  > 假设现在我们要增加`LinuxButton`，需要做的是什么？
  > 
  > 1. 写一个`LinuxButton`类继承`Button`
  > 
  > 2. 写一个`LinuxFactory`类继承`ButtonFactory`
  > 
  > 3. 客户端使用时传入`LinuxFactory`
  > 
  > 我们并没有修改`ButtonFactory`的代码，这就是开闭原则的体现：新增功能时，只通过增加新代码来实现，而不去修改旧代码。

- **缺点**：类的数量成倍增加（一个产品对应一个工厂，类的数量=产品数量*2*）

```cpp
#include <iostream>
#include <memory>

// -------------- 产品部分 --------------
class Button {
public:
    virtual ~Button() = default;
    virtual void draw() const = 0;
};

class WinButton : public Button {
public:
    void draw() const override { std::cout << "Windows 按钮\n"; }
};

class MacButton : public Button {
public:
    void draw() const override { std::cout << "Mac 按钮\n"; }
};

// -------------- 工厂部分 --------------
// 抽象工厂接口
class ButtonFactory {
public:
    virtual ~ButtonFactory() = default;
    // 纯虚函数，这就是“工厂方法”
    virtual std::unique_ptr<Button> createButton() const = 0; 
};

// 具体工厂：Windows 工厂
class WinFactory : public ButtonFactory {
public:
    std::unique_ptr<Button> createButton() const override {
        return std::make_unique<WinButton>();
    }
};

// 具体工厂：Mac 工厂
class MacFactory : public ButtonFactory {
public:
    std::unique_ptr<Button> createButton() const override {
        return std::make_unique<MacButton>();
    }
};

// -------------- 客户端使用 --------------
int main() {
    // 假设根据系统配置，我们实例化了对应的工厂
    std::unique_ptr<ButtonFactory> factory = std::make_unique<WinFactory>();

    // 客户端只管调工厂方法，根本不知道具体创建了什么 Button
    auto myButton = factory->createButton();
    myButton->draw();

    return 0;
}
```

## 三、AbstractFactory抽象工厂模式（23种设计模式之一）

- 假设我们在做跨平台 UI 框架，不仅需要`Button`（按钮），还需要`Checkbox`（复选框）、下拉框等一大堆控件。如果用工厂方法模式来创建，那么我们需要创建很多很多的类，且类的名字又很相似，这就需要调用方在写代码时小心翼翼。抽象工厂模式就是用来解决这类问题的。

- **核心思想**：抽象工厂提供一个创建一系列相关或相互依赖的接口（抽象工厂），而无需指定它们具体的类。
  
  > 简单来说：
  > 
  > - 工厂方法：一个工厂只负责一种产品（1对1）
  > 
  > - 抽象工厂：一个工厂负责一族产品（1对N）。例如`WinFactory`不仅生产`WinButton`，还必须生产`WinCheckBox`

- **优点**：
  
  - 确保产品族的一致性，防止调用方拿到错误的组合
  
  - 解耦。客户端代码`renderUI()`面向的只是`GUIFactory`、`Button`和`CheckBox`这三个抽象接口。即使具体产品族的UI风格变化，客户端也不需要修改任何一行代码

- **缺点**：对开闭原则的妥协（针对“横向扩展”）
  
  > 在设计模式中，开闭原则有时候是需要“博弈”的：
  > 
  > - 纵向扩展（增加产品族）：很爽。比如要增加一套`Linux`风格的UI，完全不需要修改旧代码，符合`COP`
  > 
  > - 横向扩展（增加新产品）：很痛苦。比如说现在产品升级，需要增加一个`Slider`滑动条，我们不仅需要增加`WinSilder`、`MacSlider`，还必须去修改`GUIFactory`这个抽象工厂，还要修改`WinFactory`、`MacFactory`等具体工厂，这违背了`OCP`

```cpp
#include <iostream>
#include <memory>

// ==========================================
// 第一部分：抽象产品族 (两个接口)
// ==========================================
class Button {
public:
    virtual ~Button() = default;
    virtual void draw() const = 0;
};

class Checkbox {
public:
    virtual ~Checkbox() = default;
    virtual void draw() const = 0;
};

// ==========================================
// 第二部分：具体产品族 (Win 系列和 Mac 系列)
// ==========================================
class WinButton : public Button {
public:
    void draw() const override { std::cout << "  [Win] 绘制方形按钮\n"; }
};

class WinCheckbox : public Checkbox {
public:
    void draw() const override { std::cout << "  [Win] 绘制带阴影的复选框\n"; }
};

class MacButton : public Button {
public:
    void draw() const override { std::cout << "  [Mac] 绘制圆角按钮\n"; }
};

class MacCheckbox : public Checkbox {
public:
    void draw() const override { std::cout << "  [Mac] 绘制极简风格复选框\n"; }
};

// ==========================================
// 第三部分：抽象工厂 (定义一族产品的创建接口)
// ==========================================
class GUIFactory {
public:
    virtual ~GUIFactory() = default;
    virtual std::unique_ptr<Button> createButton() const = 0;
    virtual std::unique_ptr<Checkbox> createCheckbox() const = 0; // 新增！
};

// ==========================================
// 第四部分：具体工厂 (保证产出的产品风格一致)
// ==========================================
class WinFactory : public GUIFactory {
public:
    std::unique_ptr<Button> createButton() const override {
        return std::make_unique<WinButton>();
    }
    std::unique_ptr<Checkbox> createCheckbox() const override {
        return std::make_unique<WinCheckbox>(); // Win 工厂必定产 Win 复选框
    }
};

class MacFactory : public GUIFactory {
public:
    std::unique_ptr<Button> createButton() const override {
        return std::make_unique<MacButton>();
    }
    std::unique_ptr<Checkbox> createCheckbox() const override {
        return std::make_unique<MacCheckbox>(); // Mac 工厂必定产 Mac 复选框
    }
};

// ==========================================
// 第五部分：客户端使用 (极度优雅)
// ==========================================
// 客户端代码只依赖抽象接口，完全不关心具体是什么风格
void renderUI(const GUIFactory& factory) {
    // 只要传进来的工厂是同一个，产出的产品族绝对搭配！
    auto btn = factory.createButton();
    auto chk = factory.createCheckbox();

    std::cout << "渲染 UI 组件:\n";
    btn->draw();
    chk->draw();
}

int main() {
    std::cout << "--- 客户端要求 Windows 风格 ---\n";
    WinFactory winFactory;
    renderUI(winFactory); // 传 Win 工厂

    std::cout << "\n--- 客户端要求 Mac 风格 ---\n";
    MacFactory macFactory;
    renderUI(macFactory); // 传 Mac 工厂

    return 0;
}
```

## 总结

| 模式                   | 核心动作               | 工厂 vs 产品 (数量关系)        | 开闭原则 (OCP) 表现                                                 | 适用场景                                                                                                                                |
|:-------------------- |:------------------ |:----------------------:|:-------------------------------------------------------------:|:----------------------------------------------------------------------------------------------------------------------------------- |
| **简单工厂** *(非标准 GoF)* | 根据传入参数内部判断创建产品     | **1** 个工厂 vs **N** 种产品 | ❌ **不符合** <br><small>*(加新产品必须改工厂的 `if-else`)*</small>         | 产品极少，且未来几乎不会变化的简单场景                                                                                                                 |
| **工厂方法**             | 子类决定实例化哪种具体产品      | **N** 个工厂 vs **1** 种产品 | ✅ **完全符合** <br><small>*(加新产品只需新增对应工厂类)*</small>               | 只有一种产品，但未来会频繁增加不同版本/实现的产品。                                                                                                          |
| **抽象工厂**             | 子类决定实例化哪**一族**相关产品 | **N** 个工厂 vs **M** 种产品 | ⚠️ **半符合** <br><small>*(纵向加“族”符合 ✅ <br> 横向加“类型”违背 ❌)*</small> | 产品分门别类（例如UI系统里有多种控件），且必须**强绑定**使用（意思是这些不同类别的产品不能随便混搭使用，它们必须来自同一个族。例如在Mac环境下new一个Windows按钮；再例如用MySQL的协议连上MySQL数据库而用PostgreSQL的语句去查询） |
