# Builder建造者模式

## 为什么需要建造者模式？

假设我们在写一个电脑组装程序，一台电脑有 CPU、GPU、RAM、硬盘等十几个部件，而且很多部件是**可选的**。

- **反面教材**（不用建造者）
  
  ```cpp
  class Computer {
  public:
      // 必选组件
      Computer(std::string cpu, std::string gpu) { ... }
      // 加个内存
      Computer(std::string cpu, std::string gpu, std::string ram) { ... }
      // 加个硬盘，但不要内存？
      Computer(std::string cpu, std::string gpu, std::string hardDrive) { ... }
      // 全都要
      Computer(std::string cpu, std::string gpu, std::string ram, std::string hardDrive) { ... }
      // ... 参数爆炸，被称为 "伸缩构造函数反模式"
  };
  ```

- **痛点**：
  
  1. 客户端调用时极其痛苦：`Computer pc("i9", "RTX4090", "", "1TB");` —— 谁能看出中间那个空字符串是RAM
  
  2. **无法保证不可变性**：如果用 `set` 方法一步步设置，对象在构建过程中处于“半成品”状态，可能被外部错误修改

- **核心思想**：分步构建，最后一次性交货
  
  > **建造者模式**：将一个复杂对象的构建与它的表示分离，使得同样的构建过程可以创建不同的表示。
  > 大白话：把一大堆参数，拆成一个个清晰的步骤（链式调用），最后按下一个“建造”按钮，直接吐出一个完美、不可变的成品对象。

```cpp
#include <iostream>
#include <string>
#include <optional>

// ==========================================
// 产品类
// ==========================================
class Computer {
private:
    // 1. 构造函数设为 private，防止外部直接 new 出半成品
    std::string cpu_;
    std::string gpu_;
    std::optional<std::string> ram_;   // 使用 optional 表示可选参数
    std::optional<std::string> drive_;

    // 只有 Builder 是它的好朋友，能调用这个构造函数
    friend class ComputerBuilder; 

    Computer(std::string cpu, std::string gpu) 
        : cpu_(std::move(cpu)), gpu_(std::move(gpu)) {}

public:
    // 只提供 getter，不提供 setter，保证对象创建后绝对不可变
    std::string getCPU() const { return cpu_; }
    std::string getGPU() const { return gpu_; }
    std::string getRAM() const { return ram_.value_or("未配置"); }
    std::string getDrive() const { return drive_.value_or("未配置"); }

    void showConfig() const {
        std::cout << "电脑配置单:\n"
                  << "  CPU: " << cpu_ << "\n"
                  << "  GPU: " << gpu_ << "\n"
                  << "  RAM: " << getRAM() << "\n"
                  << "  硬盘: " << getDrive() << "\n\n";
    }
};

// ==========================================
// 建造者类
// ==========================================
class ComputerBuilder {
private:
    // 内部维护一个半成品状态
    std::string cpu_;
    std::string gpu_;
    std::optional<std::string> ram_;
    std::optional<std::string> drive_;

public:
    // 必选组件放在构造函数里
    ComputerBuilder(std::string cpu, std::string gpu)
        : cpu_(std::move(cpu)), gpu_(std::move(gpu)) {}

    // 可选组件用链式调用 (返回自身的引用)
    ComputerBuilder& setRAM(std::string ram) {
        ram_ = std::move(ram);
        return *this; // 关键：实现链式调用的魔法
    }

    ComputerBuilder& setDrive(std::string drive) {
        drive_ = std::move(drive);
        return *this;
    }

    // 最终的“交货”按钮：组装并返回成品
    Computer build() {
        Computer pc(cpu_, gpu_);
        pc.ram_ = std::move(ram_);
        pc.drive_ = std::move(drive_);
        return pc; 
        // C++17 保证：这里绝对不会发生拷贝，而是直接把 pc 移动/构造到外部的接收者身上
    }
};

// ==========================================
// 客户端使用 (极度优雅)
// ==========================================
int main() {
    // 场景1：只要 CPU 和 GPU 的学生
    Computer pc1 = ComputerBuilder("i5", "GTX 1660").build();
    pc1.showConfig();

    // 场景2：顶配土豪 (享受丝滑的链式调用，参数顺序无所谓)
    Computer pc2 = ComputerBuilder("i9", "RTX 4090")
                       .setRAM("64GB DDR5")
                       .setDrive("2TB NVMe SSD")
                       .build();
    pc2.showConfig();

    // 场景3：只要硬盘，不要内存的奇葩需求
    Computer pc3 = ComputerBuilder("R7", "RX 7900XTX")
                       .setDrive("4TB 机械硬盘")
                       .build();
    pc3.showConfig();

    return 0;
}
```

- **代码解析**：
1. **友元突破私有构造**：`Computer`把构造函数私有化，彻底杜绝了外界`Computer c("i5", "4090")`这种不安全的直接创建。只有被授权的`ComputerBuilder`能把它造出来。

2. **`std::optional`处理可选参数**：比传空字符串 `""` 高级得多，语义明确：“要么有值，要么没配置”。

3. **`return *this`**：这是方法链的基础。每次设置完返回自己的引用，才能实现`.setA().setB().setC()`这种写法。

4. **`build()`中的按值返回**：在 C++17 之前，这里可能需要`std::move(pc)`。但在 C++17 之后，由于**保证拷贝消除**，直接`return pc;`编译器会直接在调用方的内存上构造`pc`，零拷贝零移动，极致性能。
- **优点**：
  
  1. 对象一旦`build()`出来，就只能读，不能改，杜绝了状态不一致的bug
  
  2. 极佳的代码可读性

- **缺点**：会产生额外的 Builder 类。如果产品的参数非常少（比如只有2个必选参数），用建造者就显得过度设计了，直接用构造函数就行。

## 什么情况下使用建造者模式？

不要为了用模式而用模式。如果对象只有2~3个强相关的必选参数，直接写构造函数是最好的。**建造者模式专门对付那种“属性有很多个，且多数是可选”的巨型对象。**
