# Observer观察者模式

## 为什么需要观察者模式？

假设我们在写一个气象监测系统。有一个`WeatherData`（气象站）对象，负责获取温度、湿度。  
现在有三个显示屏：

1. 手机App - 当前天气屏

2. 电脑端 - 统计预测屏

3. 报警器 - 极端天气预警屏
- **反面教材**（强耦合）
  
  ```cpp
  class WeatherData {
  public:
      void setMeasurements(int temp, int humidity) {
          // 每次数据更新，气象站必须亲自去调用所有的显示屏
          phoneApp.update(temp, humidity); 
          pcScreen.update(temp, humidity); 
          alarm.update(temp, humidity); 
      }
  };
  ```

- **痛点**：
  
  1. 违反开闭原则：如果明天老板说要加一个“大屏幕广告牌”，那么我们必须修改`WeatherData`的代码，加上`adBoard.update()`
  
  2. 气象站居然要知道所有显示屏的存在！明明是显示屏依赖气象站的数据

- **核心思想**：“我广播，你来听（发布-订阅机制）”
  
  > 观察者模式：定义对象间的一种一对多的依赖关系，当一个对象的状态发生改变时，所有依赖于它的对象都会得到通知并被自动更新。
  > 
  > - 在前面气象站的例子中，用大白话来讲就是：气象站（Subject/被观察者）只管吆喝（通知），至于谁在听（Observer/观察者）、听见了干嘛，气象站统统不管。
  > 
  > - 再举一个例子：B站UP主（被观察者）发布了一个视频（状态改变），粉丝（观察者）点了个“订阅”。UP主发视频时，绝对不会挨个给粉丝发私信说“我发视频了”，而是点一下“发布”（广播），所有订阅了他的粉丝，动态列表里自动会出现新视频。

```cpp
#include <iostream>
#include <vector>
#include <memory>
#include <string>

// ==========================================
// 1. 观察者接口
// ==========================================
class Observer {
public:
    virtual ~Observer() = default;
    // 当收到通知时，气象站会把数据推给这个函数
    virtual void onNotify(int temp, int humidity) = 0;
};

// ==========================================
// 2. 被观察者接口
// ==========================================
class Subject {
public:
    virtual ~Subject() = default;
    virtual void attach(std::shared_ptr<Observer> obs) = 0;
    virtual void detach(std::shared_ptr<Observer> obs) = 0;
    virtual void notify() = 0;
};

// ==========================================
// 3. 具体被观察者：气象站
// ==========================================
class WeatherData : public Subject {
private:
    int temperature_;
    int humidity_;
    // 💡 核心魔法：用 weak_ptr 存储观察者！
    // 这样不会增加引用计数，不会阻止观察者被正常销毁
    std::vector<std::weak_ptr<Observer>> observers_;

public:
    void setMeasurements(int temp, int humidity) {
        temperature_ = temp;
        humidity_ = humidity;
        notify(); // 数据一变，立刻广播
    }

    void attach(std::shared_ptr<Observer> obs) override {
        observers_.push_back(obs); 
    }

    void detach(std::shared_ptr<Observer> obs) override {
        // 从 vector 中移除指定的观察者（C++20写法）
        std::erase_if(observers_, [obs](const std::weak_ptr<Observer>& wp) {
            return wp.lock() == obs;
        });
    }

    void notify() override {
        std::cout << "\n--- 气象站广播通知 ---\n";
        // 💡 遍历时的安全操作
        for (auto it = observers_.begin(); it != observers_.end(); /* 不在这里++ */) {
            // 尝试将 weak_ptr 提升为 shared_ptr
            if (auto sp = it->lock()) {
                sp->onNotify(temperature_, humidity_);
                ++it;
            } else {
                // 💥 绝妙之处：如果 sp 提升失败，说明观察者已经被销毁了！
                // 直接顺手把这个无效的 weak_ptr 从列表里清理掉，防止以后报错
                it = observers_.erase(it);
                std::cout << "[系统清理] 发现一个已销毁的观察者，已自动移除。\n";
            }
        }
    }
};

// ==========================================
// 4. 具体观察者：手机 App 显示屏
// ==========================================
class PhoneDisplay : public Observer {
private:
    std::string name_;
public:
    PhoneDisplay(const std::string& name) : name_(name) {}

    void onNotify(int temp, int humidity) override {
        std::cout << "[" << name_ << " 手机APP] 收到更新: " 
                  << temp << "°C, 湿度 " << humidity << "%\n";
    }
};

// ==========================================
// 5. 客户端使用
// ==========================================
int main() {
    auto weatherStation = std::make_shared<WeatherData>();

    // 创建两个观察者，用 shared_ptr 托管
    auto app1 = std::make_shared<PhoneDisplay>("张三的iPhone");
    auto app2 = std::make_shared<PhoneDisplay>("李四的安卓");

    // 订阅（注册）
    weatherStation->attach(app1);
    weatherStation->attach(app2);

    // 气象数据更新 -> 两个 App 都会收到
    weatherStation->setMeasurements(25, 60);

    // 模拟 App1 被用户关闭了（智能指针引用归零，对象销毁）
    std::cout << "\n>>> 张三关闭了手机APP <<<\n";
    app1.reset(); 

    // 再次更新气象数据
    weatherStation->setMeasurements(28, 55);
    // 注意看输出：张三的 App 不会再收到通知，且气象站自动清理了它！

    return 0;
}
```

- **核心思想**
  
  1. **为什么不用裸指针`Observer*`？** 因为一旦观察者被`delete`，气象站不知道，继续调就会段错误。
  
  2. **为什么不用`std::shared_ptr<Observer>`？** 如果气象站持有了`shared_ptr`，那么哪怕外部想把App关掉（释放内存），因为气象站还在引用它，引用计数不为0，App**永远无法被销毁**，这就叫**内存泄漏**（逻辑上的）。
  
  3. **`weak_ptr`的完美平衡**：它就像一张“借条”。气象站只拿着借条，不拥有观察者。要通知时，拿借条去换(`lock()`)，如果对象还在就通知；如果对象已经被外面销毁了，换借条失败(`lock()` 返回空)，直接顺手把借条撕了(`erase`)。**内存绝对安全，且自动垃圾回收！**

- **优点**：
  
  1. **极度解耦**。被观察者完全不认识观察者，只依赖抽象接口。
  
  2. **动态通信**。可以在运行时随意增加或删除观察者，无需修改任何代码。

- **缺点**：
  
  1. **性能/时间开销**。如果观察者极多，每次广播遍历列表会耗时有延迟。
  
  2. **调试困难**。因为广播是隐式的，有时候很难追踪到底是哪个观察者触发了bug。
  
  3. **循环依赖风险**。如果A观察B，B又观察A，会陷入死循环广播（需要业务上规避）。

## 扩展：推模式与拉模式

上面的代码是**推模式**：气象站主动把`temp, humidity`塞给`onNotify(temp, humidity)`

- **缺点**：不管观察者需不需要湿度，都硬塞给它



还有一种是**拉模式**：`onNotify(Subject* who)`只传一个气象站的指针过去

- **优点**：观察者拿到指针后，自己调用`who->getTemp()`，按需索取
- **缺点**：观察者必须依赖具体的`WeatherData`类，耦合度稍微回升了一点点
